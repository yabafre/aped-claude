# Step 8: Ticket System Setup (with sync-log auditability)

## MANDATORY EXECUTION RULES (READ FIRST):

- 🛑 If `ticket_system: none`, skip this step entirely (record `ticket_sync: skipped` under `phases.epics`)
- 🛑 Wrap every provider call with sync-log calls (start / phase / record / end)
- 🛑 GATE: user confirms before any ticket is created
- 🚫 NEVER overwrite ticket bodies on re-sync — append to `modified_tickets`

## CONTEXT BOUNDARIES

- `epics.md` written and committed in working state.
- `state.yaml` updated.

## YOUR TASK

Read `ticket_system` from config. If `none`, skip and mark `ticket_sync: skipped`. Otherwise, walk the 7-step sync flow with structured audit logs.

Read `ticket_system` from `{{APED_DIR}}/config.yaml`. If `none`:

- Skip this step entirely.
- Set `phases.epics.ticket_sync: skipped` in `state.yaml`.

## STEP 8.0 — OPEN THE SYNC LOG

Before any provider call, capture an audit-log path. The provider name is the value of `ticket_system` from config (`linear` | `github-issues` | `gitlab-issues` | `jira`); pass it verbatim:

```bash
LOG=$(bash {{APED_DIR}}/scripts/sync-log.sh start <provider>)
```

Capture `$LOG` once and reuse it for every subsequent `phase` / `record` / `end` call. Surface the path to the user at the end.

If `sync_logs.enabled: false` in `config.yaml`, the helper exits 0 silently with empty stdout — `$LOG` will be empty and downstream calls will be silent no-ops.

## STEP 8.1 — CHECK CREDENTIALS

Verify the CLI/auth is configured:

- `github-issues`: `gh auth status`
- `gitlab-issues`: `glab auth status`
- `linear`: `LINEAR_API_KEY` env var or linear CLI
- `jira`: `JIRA_BASE_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN` env vars

If not configured: warn the user with setup instructions and continue **without** ticket sync. Mark `ticket_sync: skipped` in state.yaml. Skip to step 09.

After the auth check completes:

```bash
bash {{APED_DIR}}/scripts/sync-log.sh phase $LOG auth_check complete
bash {{APED_DIR}}/scripts/sync-log.sh record $LOG api_calls_total 1
```

## STEP 8.2 — CHECK PROJECT

Verify the target project exists:

- `github-issues`: `gh repo view`.
- `gitlab-issues`: `glab repo view`.
- `linear`: ask the user for the team key (e.g., `TEAM`) — store in config.
- `jira`: ask the user for the project key — store in config.

After the project resolution completes:

```bash
bash {{APED_DIR}}/scripts/sync-log.sh phase $LOG projects complete '{"calls":N,"created":[...]}'
bash {{APED_DIR}}/scripts/sync-log.sh record $LOG api_calls_total N
```

## STEP 8.3 — PREVIEW TO USER (GATE)

Show the user what will be created:

```
Will create in {ticket_system}:
  📦 Milestone: Epic 1 — Users can manage inventory
     🆕 Issue: [S] 1-1-project-setup
     🆕 Issue: [M] 1-2-inventory-crud
     ...
```

⏸ **GATE: User confirms before creating anything in the ticket system.**

## STEP 8.4 — CREATE MILESTONES (one per epic)

Title: `Epic {N}: {epic title}`. Description: epic summary + FR coverage list.

- **github-issues**: `gh api repos/{owner}/{repo}/milestones -f title="Epic 1: ..." -f description="..."`
- **gitlab-issues**: `glab api projects/{id}/milestones -f title="Epic 1: ..." -f description="..."`
- **linear**: project/cycle API via curl or linear CLI.
- **jira**: epic issue type (JIRA has native epics).

After the milestone batch:

```bash
bash {{APED_DIR}}/scripts/sync-log.sh phase $LOG milestones complete '{"calls":M,"created":{...}}'
bash {{APED_DIR}}/scripts/sync-log.sh record $LOG api_calls_total M
```

## STEP 8.5 — CREATE ISSUES (one per story)

Title: `[{size}] {story-key}: {story title}` (e.g., `[M] 1-2-inventory-crud: CRUD for inventory items`).

Description (markdown body):

```markdown
## User Story
As a {role}, I want {capability}, so that {benefit}.

## Acceptance Criteria
- [ ] AC-1: Given ... When ... Then ...

## Covered FRs
- FR-1: {FR title from PRD}

## UX References (if UX exists)
- Screen: {screen name}
- Components: {component list}

## Estimated Size
{S | M | L}

---
Generated by APED Method v{{CLI_VERSION}}
```

**Labels:**
- Type label: `🆕 feature` / `🔄 refactor` / `🔁 update`.
- Size label: `size/S` / `size/M` / `size/L`.
- Epic label: `epic/{N}`.
- Phase label: `aped/story`.

**Milestone:** assign to the epic milestone created in 8.4.

After the issue creation batch:

```bash
bash {{APED_DIR}}/scripts/sync-log.sh phase $LOG labels complete '{"calls":L,"created":{...}}'
bash {{APED_DIR}}/scripts/sync-log.sh record $LOG api_calls_total L
bash {{APED_DIR}}/scripts/sync-log.sh phase $LOG issues_created complete '{"calls":I,"issues":[...]}'
bash {{APED_DIR}}/scripts/sync-log.sh record $LOG api_calls_total I
```

If any tickets were modified mid-sync (re-titled, re-described, project moved), record them under a `modified_tickets` phase. If any tickets were moved to the future-scope bucket (M2), record them under `out_of_scope_moves`.

## STEP 8.6 — STORE TICKET IDS

Update `{{OUTPUT_DIR}}/epics.md` — add a `**Ticket:** {ticket-id}` line under each story.

Update `{{OUTPUT_DIR}}/state.yaml`:

```yaml

  stories:
    1-1-project-setup:
      status: pending
      ticket: "#42"
    1-2-inventory-crud:
      status: pending

```

## STEP 8.7 — CLOSE LOG + WRITE STRUCTURED STATE

```bash
bash {{APED_DIR}}/scripts/sync-log.sh end $LOG
```

Surface the log path to the user (*"Sync complete — audit trail at `docs/sync-logs/<provider>-sync-<ISO>.json`"*).

Now write the **top-level `ticket_sync` block** to `{{OUTPUT_DIR}}/state.yaml`. The shape is provider-agnostic — fields below are mandatory; provider-specific detail goes inside `projects` and `milestones`:

```yaml
ticket_sync:
  provider: "<linear | github | gitlab | jira>"
  sync_id: "<basename of $LOG with .json stripped>"
  synced_at: "<ISO 8601 now>"
  sync_log: "docs/sync-logs/<provider>-sync-<ISO>.json"
  directive_version: "<from sync-log if available, else null>"
  projects:
    # Provider-specific shape. Linear example:
    foundation:
      id: "<linear-project-id>"
      name: "[M1] Foundation"
      lead: "<email>"
      target_date: "<ISO date | null>"
  milestones:
    "foundation/epic-0": "<milestone-id>"
  modified_tickets:
    - id: "<ticket-id>"
      fields: ["title", "description", "project"]
      reason: "<one-liner>"
      original_description_sha256: "<sha256 of pre-sync body>"
      labels_added: ["<label1>"]
      project_moved: "<from> → <to>"
  totals:
    api_calls_total: <int>
```

If M2 / future-scope buckets exist, ALSO write a top-level `backlog_future_scope` block with `{ project_id, tickets: [{id, category}] }`.

Update `phases.epics` in state.yaml with the structured fields:

```yaml
pipeline:
  phases:
    epics:
      status: "done"
      output: "{{OUTPUT_DIR}}/epics.md"
      completed_at: "<ISO 8601 now>"
      epic_count: <int>
      story_count: <int>
      fr_coverage: "<M>/<N> [optional descope note]"
      ticket_sync: "synced"   # synced | skipped | failed
      synced_at: "<ISO 8601 now>"
```

If `ticket_system: none` (or auth check failed and we skipped sync), set `ticket_sync: skipped` and omit the top-level `ticket_sync` block entirely.

Re-syncs append to `modified_tickets` (never overwrite); other fields are replaced with the new run's values.
